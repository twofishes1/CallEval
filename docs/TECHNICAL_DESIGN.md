# 技术设计说明：图谱建模、路径生成与评分方法

本文档面向开发者与评审人员，详细说明 Eval1 系统中 **Layer1 规则知识图谱建模**、**自适应路径生成**、**Layer2 对话模拟**以及 **Layer3 评分**四个核心模块的设计原理与实现逻辑。

各模块的实现代码分布如下：图谱建模位于 `eval1/layer1/rule_graph.py`；路径生成涉及 `path_enumerator.py`、`path_coverage_builder.py`、`path_coverage_planner.py`、`path_linear_curator.py`、`path_plan_config.py` 及 `pipeline/plan_compat.py`；Layer2 对话仿真的核心在 `eval1/layer2/simulation_graph.py`；评分方法分布在 `eval1/layer3/rule_judge.py`、`llm_judge.py`、`aggregator.py` 和 `rubrics.py`。

---

## 一、Layer1 规则知识图谱建模

规则知识图谱（Rule Knowledge Graph，RuleKG）的核心目标是将人工编写的自然语言指令转化为机器可遍历的有向图结构，从而支持系统地枚举所有有意义的对话路径（覆盖主干流程、条件分支、FAQ、挽留等场景），明确每条路径上激活的约束规则集合以供 Layer3 评分使用，并在前端可视化图谱结构辅助人工审核。图谱以 NetworkX 有向图实现，入口类为 `RuleGraphBuilder`。

![Layer1 规则知识图谱结构示意](C:\Users\vis\.cursor\projects\e-contest2/assets/layer1_rule_graph_structure.png)

---

### 1.1 图谱结构与节点类型

图谱由四个层次叠加构成：流程主干层、条件分支层、约束挂载层和全局守卫层。图中的节点按功能分为七种类型。系统级元节点（`meta`）包括 START、END、GLOBAL\_DIALOGUE 和 GLOBAL\_BOUNDARY，代表对话的起止以及全局约束汇总。流程步骤节点（`flow_step`）对应指令 Call Flow 中的有序步骤，编号为 F1、F2、……Fn，每步保留原始文本作为节点标签。转移节点（`transition`）代表对话可能发生的场景切换，包括 CLOSING（收尾）、FAQ\_NORMAL（合规追问）、FAQ\_OOB（越界追问）、OBJECTION（用户拒绝）、F3\_RETAIN（挽留）和 OBJ\_FINAL（最终拒绝关闭）；其中 F3\_RETAIN 与 OBJ\_FINAL 仅在指令包含「挽留不想配送的骑手」类语义时才创建。条件分支节点（`flow_branch`）对应某流程步骤下来自指令原文"若…则…"语句的条件分支，操作子步骤节点（`op_step`）是分支内部的多步操作链。此外还有约束节点（`constraint`，标准 ID 如 D1、K1、B1、R1）和知识要点节点（`knowledge`），分别来自指令的约束字段和 FAQ 段。

---

### 1.2 流程主干层构建规则

建图的第一步是创建所有固定的元节点和转移节点，随后依次将指令 Call Flow 的每个步骤映射为 F 编号节点，形成有序的主干链：START → F1 → F2 → … → Fn → CLOSING → END。所有顺序边的置信度为 1.0，末步到 CLOSING 的边置信度略降至 0.95，以体现收尾环节的不确定性。若指令不含任何 flow\_steps，图退化为 START → CLOSING → END 的最简结构。

在主干之上，系统为每个流程步 Fi 添加三条中断出边，反映用户在任意步骤均可能偏离主干的现实：用户表示拒绝时跳转到 OBJECTION（置信度 0.75），用户合规追问时跳转到 FAQ\_NORMAL（置信度 0.80），用户提出越界问题时跳转到 FAQ\_OOB（置信度 0.55）。START 节点也有一条到 OBJECTION 的出边（置信度 0.60），覆盖开场即拒绝的场景。回归轨方面，FAQ\_NORMAL 处理完毕后通过 `resume_before:Fi` 边返回被打断的步骤继续推进，而 FAQ\_OOB 不进行回答而是直接通过 `redirect_and_close` 边跳到 CLOSING，体现越界问题应收束而非深入的处理原则。

对于含挽留语义的指令，图中还会添加挽留轨：START 和各 Fi 节点均可通过 `retention_jump` 边（触发条件为用户早期或中途拒绝）跳入 F3\_RETAIN 节点，F3\_RETAIN 依据用户后续反应决定返回流程继续（`resume_before:Fi`）或进入 OBJ\_FINAL（用户仍然拒绝），OBJ\_FINAL 最终通过顺序边到达 END。

---

### 1.3 条件分支层构建规则

条件分支从指令原文中确定性解析，全程不依赖 LLM。系统优先使用结构化解析器（`parse_instruction_branches`），识别形如"若[骑手表示有顾虑]→[解释说明]"的结构化条件分支，提取条件、动作、目标步骤和操作子步骤等字段；对未被结构化解析器识别的"若…则…"段落，使用正则兜底解析器（`extract_branches_from_block`）补充覆盖。

分支层最关键的设计原则是：当某步 Fi 存在条件分支时，系统会删除 Fi 到 F(i+1) 的直连顺序边，迫使对话必须经过分支节点才能继续推进，而不能绕过分支直接前进。这保证了所有分支覆盖都是强制的。若某分支内含有多个操作子步骤，则展开为 op 节点链（按 op::step::branch::i 命名），顺序相连后 goto 到目标步骤或下一步。

---

### 1.4 约束挂载层与全局守卫层

约束节点按类型挂载到对应的图状态节点上，反映约束在哪种场景下被激活。流程约束（FLOW 类型）通过自环边挂载到对应流程步骤，表示该约束随流程执行持续生效；知识约束（KNOWLEDGE 类型）通过 `on_user_ask` 边挂载到 FAQ\_NORMAL，表示当用户提问时才触发相应知识义务；边界约束（BOUNDARY 类型）挂载到 FAQ\_OOB，在越界问题场景下强制守界；话术约束（DIALOGUE 类型）挂载到 CLOSING，并在全程每次 Bot 回复时持续检查。指令 FAQ 段的知识要点节点（`knowledge_nodes`）同样以 `on_user_ask` 方式挂到最适合的状态节点，默认为 FAQ\_NORMAL。

在约束挂载层之上，还有两个全局守卫元节点：GLOBAL\_DIALOGUE 汇总所有话术约束，GLOBAL\_BOUNDARY 汇总所有边界约束，均通过 `global_guard` 边连接。这两个节点为 Layer3 评分提供全局约束集合的索引入口，使评分器无需遍历全图即可快速获取全部话术约束和边界约束的集合。

---

### 1.5 冲突检测

图构建完成后，系统执行一次确定性的静态冲突检测，不调用 LLM。目前检查两类问题：若两条 DIALOGUE 类约束的文本内容相同，则报告 `duplicate_dialogue_constraint` 冲突；若流程节点有两步及以上但 F1 到 F2 的顺序边不存在，则报告 `flow_sequence_break`（严重级别为 critical）。检测结果写入图谱 artifact，供前端展示和人工审查。

## 二、自适应路径生成

路径生成要回答的核心问题是：对于一部包含多个条件分支、FAQ 追问、挽留等逻辑的复杂对话指令，应该测哪些路径？全量路径排列在分支较多时会产生组合爆炸，完全随机又无法保证覆盖。Eval1 采用自适应结构化覆盖策略：先根据图谱特征推断指令的任务模式，再按对应策略在路径数量上限（默认 64 条）内最大化有效覆盖。

![Layer1 自适应路径生成流程](C:\Users\vis\.cursor\projects\e-contest2/assets/layer1_adaptive_path_generation.png)

---

### 2.1 任务模式推断

路径生成的入口是 `infer_path_plan_config`，该函数对图谱做结构检测——这是一个完全确定性的过程，不调用 LLM——并输出 `PathPlanConfig`。检测逻辑如下：若图谱中存在 `branch::` 前缀的条件分支节点，则判断为 `branched` 模式，FAQ 变体将附着在 F2/F3/F4/F5/F7 等关键步骤，OOB 变体主要附着在 F4；若图谱中存在 F3\_RETAIN 节点（含挽留轨），则判断为带挽留变体的线性模式（`linear + include_retention_variants`），FAQ/OOB 附着步骤集中在 F2/F3/F4；否则退化为纯线性模式，FAQ 和 OOB 变体附着在除 F1 以外的全部步骤。分支任务和挽留任务不共用策略，因为两者的路径空间结构截然不同。

---

### 2.2 结构化覆盖（Branched 模式）

分支模式下，核心路径由 `build_minimal_coverage_paths` 生成，底层依赖 `walk_path` 函数。`walk_path` 接受一个分支选择字典（将步骤编号映射到分支 ID），从 START 出发沿图行走到 END，遇到分支步骤时按字典选择对应分支，跟随操作子步骤链后 goto 到目标步骤，最终返回完整的节点序列。

路径生成采用三级递进覆盖策略，逐级补充直到满足全分支覆盖。第一级是核心分支组合网格：对指令中覆盖用户意向与配送能力的关键步骤（以 instruction\_2 为例是 F4 和 F6）做两维笛卡尔积，每种组合对应一条路径，共约 8 条，确保高优先级分支节点全部出现。第二级是替代分支覆盖：对其他步骤（F1、F2、F3、F5）中非默认的备选分支，与 F6 的各个变体分别组合，补充覆盖那些在主流配合路径下不会触发的边缘分支场景。第三级是兜底覆盖：统计前两级已覆盖的所有 `branch::` 节点集合，对仍未出现在任何路径中的分支单独构造一条路径补入，保证每个条件分支节点至少在一条路径中出现。

---

### 2.3 叠加变体路径

无论何种模式，系统都会在结构化主干路径的基础上叠加三类变体路径，去重后并入路径集合。

**FAQ 变体**在基准路径某步 Fi（或该步最后一个 branch/op 节点）之后插入 FAQ\_NORMAL 节点，再通过 `resume_before:Fi` 返回，模拟用户在该步完成后追问的场景。附着的步骤由 `PathPlanConfig.faq_attach_steps` 控制，不同模式有不同的锚点集合。

**OOB 变体**在基准路径某步 Fi 之前插入 FAQ\_OOB 节点，模拟用户在该步开始前提出越界问题，路径随即跳转到 CLOSING 结束，测试 Bot 守界并礼貌收束的能力。

**探针变体**在基准路径中插入场景探针节点（PROBE\_D9\_BUSY 模拟骑手反映太忙，PROBE\_D10\_DRIVE 模拟骑手正在开车），专门用于测试对应的特殊场景约束是否被正确处理。

---

### 2.4 线性 + 挽留模式：DFS 策展

对于含挽留轨的线性指令，系统采用 DFS 策展机制处理用户可能在不同步骤拒绝的场景多样性。首先在图上做深度优先搜索，收集所有包含 F3\_RETAIN、FAQ\_NORMAL、FAQ\_OOB、OBJ\_FINAL 等节点的路径组合作为候选池；随后由 `curate_retention_flow_paths` 按语义模板从候选池中筛选代表性路径，覆盖五类模板：某步后纯 FAQ（无挽留/OOB）、某步后纯 OOB（无挽留/FAQ）、进入挽留后继续流程（F3\_RETAIN → 返回 F4）、先 FAQ 后挽留再继续、先 OOB 后挽留再继续。策展结果最终与第一级主干路径合并，按路径语义指纹去重后截断到 `max_paths`。分支任务跳过 DFS 阶段，仅使用结构化网格覆盖。

---

### 2.5 语义去重机制

路径去重采用语义指纹而非节点序列的精确比较。`path_dedupe_key` 提取路径的核心语义特征：FAQ 节点的附着步骤位置、是否包含 OOB 节点、探针节点集合以及所有 `branch::` 节点的有序集合。两条路径若上述特征完全相同，则视为语义等价，保留先出现的一条，丢弃后者。这避免了节点序列因遍历顺序略有差异但实际测试场景相同的冗余路径进入执行计划。

## 三、Layer2 对话模拟层设计

Layer2 接收 Layer1 生成的执行计划，驱动一次完整的多轮对话仿真，输出 Bot 行为记录与违规证据，供 Layer3 评分使用。

![Layer2 对话模拟层架构](C:\Users\vis\.cursor\projects\e-contest2/assets/layer2_design_diagram.png)

---

### 3.1 对话运行流程

Layer2 以 LangGraph StateGraph 为循环引擎，整体拓扑为：每轮开始时先由 `init_turn` 节点调用终止检查器进行预检，若未触发终止则根据当前状态路由——若消息列表为空（外呼模式首轮），直接进入 Bot 节点让 Bot 先发 Opening Line；否则进入用户模拟节点生成用户话语，再流转到 Bot 节点生成回复，Bot 节点完成后再次执行终止检查，若未终止则回到 `init_turn` 开启下一轮，形成"检查→用户→Bot→检查→…"的主循环，直到终止条件触发时跳出到 END 节点。

每轮状态以 `DialogueState` 快照形式在节点间传递，核心字段包括：对话历史消息列表（`messages`，格式为按轮次编号的 `{turn, role, content}` 条目）、GoalFSM 实例（携带路径指针与已覆盖节点集合）、累积违规列表（`violations`）、当前 Persona 类型、Bot 侧状态（`bot_state`，包含 F4 输出索引、槽位值、Bot 动作日志等）、当前轮次计数和本次仿真的轮数上限。

系统还维护了 `stalled_rounds` 计数器，用于检测对话推进停滞。若连续两轮以上 FSM 指针未发生移动且用户动作不属于主动中断类型（拒绝、追问、跑题），系统会以 `comply` 强制推进 FSM，避免对话因动作识别模糊而陷入死锁。

---

### 3.2 用户模拟器

用户模拟器的设计目标是：在每轮对话中，基于当前路径位置和 Persona 风格，生成一条语义合理、行为可控的用户话语，同时对该话语的动作类型进行检测标注，为 FSM 推进和终止判断提供依据。

**GoalFSM 路径指针驱动。** GoalFSM 由 Layer1 枚举的路径节点序列初始化，采用线性指针而非事件触发的跳转机制。每当用户动作满足当前节点的转移条件，`try_transition` 方法将指针推进到下一节点，并将新节点追加到 `covered_nodes` 集合，作为路径覆盖的证据。这种设计保证了仿真对话严格沿着预定义的路径推进，而非自由游走。

**路径动作推断。** 在生成用户话语之前，系统首先根据 GoalFSM 当前节点类型推断出本轮用户应执行的动作，作为 LLM 生成的强约束。具体规则为：当 FSM 处于异议或挽留节点（OBJECTION、F3\_RETAIN、OBJ\_FINAL）时，推断用户动作为 `reject`，用于驱动挽留和拒绝路径；处于合规追问节点（FAQ\_NORMAL）时推断为 `ask_question`；处于越界问题节点（FAQ\_OOB）时推断为 `off_topic`；处于主干流程步骤节点时推断为 `comply`，配合推进流程。质疑型（QUESTIONING）Persona 有一项特殊处理：若当前处于流程步骤节点且本步骤尚未被追问过，即便路径动作为 `comply`，系统也会将其覆盖为 `ask_question`，并附带提示"就 Bot 刚说的规则追问一句依据、后果或怎么算"，以提升 FAQ 场景的覆盖深度。

**话语生成。** `UserSimulatorAgent` 调用 `qwen-plus` 模型生成用户话语，输入上下文包括：Persona 卡（性格类型、配合度、情绪状态、禁忌行为）、推断得到的路径动作及话语提示、完整对话历史、用户历史发言、跨轮积累的 `UserContextMemory`（已知信息摘要）、当前 FSM 状态以及允许的动作集合。这些信息共同约束 LLM 输出，使生成的用户话语在行为上符合路径预期，在措辞上符合 Persona 风格。

**动作标签检测。** 对生成的用户话语，`ActionDetector` 执行两阶段动作识别：优先进行关键词匹配，词典命中时直接返回对应动作标签；若无法确定则调用 LLM 进行语义分类兜底。对置信度低于阈值的识别结果，标记 `needs_review=True`，在最终报告中提示人工复核。识别出的动作标签同时用于 FSM 转移（`try_transition`）和终止条件判断。

---

### 3.3 被测 Bot

被测 Bot 的核心职责是：接收完整指令和对话历史，在当前 FSM 状态下生成一条符合规范的回复话语，同时触发 DST 约束检测，将违规证据实时记录到状态中。Bot 节点按以下优先级顺序选择生成策略，高优先级命中后不再执行后续策略。

**Opening Line（首轮外呼）。** 若为首轮且指令包含 `opening_line` 字段，Bot 直接输出经变量替换后的 Opening Line，不调用 LLM。这确保外呼场景的首句话严格按照指令要求输出，同时记录骑手姓名是否被正确使用（`name_used`）。

**F4 配送要点拆分输出。** 若当前 FSM 节点为 F4 且检测到指令 F4 步骤包含配送三要点（排名机制、拒单规则、恶劣天气安全），系统使用 `f4_speech_index` 索引机制按顺序逐条输出，而非一次性全部说出，不调用 LLM。`sync_f4_completion` 会检查历史回复，动态跳过已完整输出的要点；三要点全部覆盖后切换到确认语输出。

**强制脚本优先。** 在调用 LLM 之前，系统检查是否存在必须输出的固定话术（如特定步骤的合规声明）。若命中则直接使用强制脚本，标记为 `mandatory_script` 动作类型，保证关键合规信息不被 LLM 改写或遗漏。

**LLM 生成及特殊节点处理。** 其余情况调用 `BotWrapper.reply`，输入包括完整指令文本（作为 System Prompt，含变量替换）、当前步骤说明文本、完整对话历史和用户动作。部分节点有专项处理策略：CLOSING 节点先推断收尾语气，LLM 回复不合格时回退到预定义模板；OBJ\_FINAL 节点（最终拒绝关闭）和 FAQ\_OOB 节点（越界问题守界）均直接使用模板话术，不调用 LLM，以保证这两类场景下回复的合规性。

**重复守卫。** 每次 Bot 生成回复后，系统通过精确匹配、前缀匹配、主题关键词多次命中等多重策略检测新回复与近期历史是否语义重复，若重复则从候选集中调用 `pick_non_repeating` 选择非重复话术替换。系统维护最近 4 条 Bot 话语作为去重窗口（`repeat_guard_window`），并将重复次数记录到 `repetitive_bot_count` 供 Layer3 扣分使用。

**DST 违规检测。** Bot 回复追加到消息列表后，立即调用 `dst.check_constraints` 进行实时约束检测。系统检查两类违规：一是话术长度超限（回复字数超过 30 字，但 Opening Line、F4 拆分输出、强制脚本豁免），触发时追加类型为 `dialogue_length` 的违规记录，扣分值为 7 分；二是边界越权（回复中出现越权承诺词，如超出授权范围的承诺），触发时追加类型为 `hard_boundary` 的违规记录，并将 `hard_violation` 状态置为 True。所有违规均携带轮次编号、约束 ID、Bot 原文和扣分值，供 Layer3 评分直接使用。

---

### 3.4 对话终止条件

终止检查在每轮的 `init_turn` 阶段（对话开始前）和 Bot 节点完成后双重执行，确保高优先级终止条件能立即停止对话而不等到下一轮。终止条件按以下优先级顺序判断，高优先级条件满足时立即终止，跳过低优先级检查。

第一优先级为**硬边界违规**（`hard_violation`）：DST 检测到边界越权时将状态标记为 True，系统立即终止对话，确保严重违规不被继续执行。

第二优先级为**挂断**（`hangup`）：检查最新一条消息是否包含挂断关键词。用户侧关键词为"挂了"、"再见"、"先这样"；Bot 侧关键词为"再见"、"感谢您的来电"、"谢谢您的来电"、"祝您"、"先这样"、"不打扰了"。任意一方说出挂断词即视为通话结束。

第三优先级为**目标达成**（`goal_achieved`）：GoalFSM 的 `is_goal_achieved()` 返回 True，表示路径中所有预期节点均已覆盖，对话任务完成。

第四优先级为**用户持续拒绝**（`user_refused`）：在 OBJ\_FINAL 节点 Bot 完成收尾话术后，系统将 FSM 推进到终态并写入 `user_refused` 终止原因，下一轮检查时触发该条件。

第五优先级（最低）为**轮数上限**（`max_turns`）：当前轮次计数达到或超过本次执行计划设定的上限时终止。若无任何条件触发，检查返回 `continue`，对话继续下一轮。

---

### 3.5 对话轮数设定

每条执行计划的最大轮数（`max_turns`）由 `estimate_plan_max_turns` 函数动态计算，而非使用固定值，目的是根据路径复杂度和 Persona 行为特征合理分配对话预算。

计算过程分四步。首先以配置中的基础缓冲值（`min_turns_buffer`）为起点，遍历路径中的每个节点（排除无对话内容的 START、END 等元节点），按节点类型累加轮次成本：每个流程步骤节点（F1 至 Fn）贡献 2 轮成本，CLOSING 节点贡献 1 轮，其余节点（如 FAQ\_NORMAL、OBJECTION 等）各贡献 1 轮。其次叠加 Persona 附加轮数，不同性格的用户在对话中产生的额外交互量不同，由 `persona_turn_extra` 配置控制。然后针对特定的场景组合进一步追加：质疑型 Persona 搭配含 FAQ 节点的路径额外加 1 轮，抵触型 Persona 搭配含挽留节点的路径额外加 2 轮，懵懂型搭配 FAQ 节点加 1 轮，F4 节点路径搭配质疑型或抵触型各加 1 轮。最后对计算结果施加上下限约束：下限取配置默认值与路径自身基准轮数中的较大值，上限取全局最大轮数绝对值（`max_turns_absolute`）。

这一动态计算机制确保了简单主干路径不会浪费过多轮数预算，而含有 FAQ 追问、挽留异议等复杂交互的路径在搭配行为更复杂的 Persona 时能获得足够的轮次空间，提升仿真质量的同时控制整体运行成本。

---

### 3.6 执行计划生成（Path × Persona 全量笛卡尔积）

执行计划由 Layer1 枚举的所有路径与全部六种 Persona 做全量笛卡尔积生成，每个路径与 Persona 的组合均对应一条独立的 `ExecutionPlan`，不做预先过滤或跳过。系统仍维护语义匹配规则，对语义上不匹配的组合（如配合型 Persona 搭配异议路径）标注 `potential_contradiction` 并记录冲突理由，但该标注仅作数据注记，不影响实际执行。这一设计保留了全面覆盖的可能性，同时为后续数据分析提供了语义对照信息；若需恢复语义过滤，可对笛卡尔积结果调用 `filter_compatible_plans` 进行二次筛选。

## 四、Layer3 评分方法

![Layer3 评分体系](C:\Users\vis\.cursor\projects\e-contest2/assets/layer3_design_diagram.png)

---

### 4.1 评估体系总览

Layer3 承接 Layer2 仿真输出的 `DialogueTrace`（对话记录、违规证据、路径覆盖率、Bot 行为状态），负责将这些结构化证据转化为可量化、可解释的评分结果。系统采用双轨评分加权汇总的架构：规则分轨（RuleJudge）依据仿真过程中实时检测到的确定性证据进行扣分计算，结果可精确复现；LLM 分轨（LLMJudge）以六维 Rubric 锚定表引导大模型对对话质量做语义评估，捕捉话术自然度、挽留策略等难以用规则量化的维度。两轨结果由 Aggregator 按固定权重合并为单条用例的总分，再跨所有执行计划聚合为数据集级别的系统综合评分。评分结果以结构化 `EvalReport` 输出，每条报告包含完整的证据链，支持逐轮追溯。

---

### 4.2 单用例评分（评分具体构成）

每条执行计划（`ExecutionPlan`）完成 Layer2 仿真后，独立经历三个评分步骤，最终输出一条 `EvalReport`。

**规则分计算（RuleJudge）。** 规则分从满分 100 分起，按照 Layer2 仿真中积累的违规记录和 Bot 行为指标逐项扣除。主要扣分项及对应扣分值如下。硬边界违规（Bot 越权承诺、超出授权范围的回复）每次扣 20 分；话术长度超限（单轮回复字数超过 30 字，Opening Line 及 F4 拆分输出豁免）每次扣 7 分；路径覆盖缺口（实际覆盖节点数低于阈值）按"(1 − 覆盖率) × 20"计算扣分；流程未完成（路径中间断开）扣 10 分；其他类型违规按记录中的 `deduction` 字段扣分，默认每次 5 分。在违规扣分之外，流程覆盖率低于 60% 时额外扣 10 分，覆盖率在 60% 至 85% 之间时额外扣 5 分；F4 配送路径中三要点（排名机制、拒单规则、恶劣天气安全）有缺失时扣 10 分；Bot 产生语义重复回复时每次扣 4 分（累计上限 15 分）；开场白与指令要求不匹配时扣 8 分。所有扣分项对同一约束 ID 与同一轮次的组合只计一次，避免重复惩罚。当硬边界违规累计超过 2 次时，触发 `hard_fail`，规则分直接归零，后续 Aggregator 将总分强制置为 0 分、等级标记为 F，不再参考 LLM 分的贡献。

**LLM 评委分计算（LLMJudge）。** LLM 评委（默认使用 `qwen-turbo`）接收以"[T轮次] 角色：内容"格式编排的完整对话记录，结合六维 Rubric 锚定表和指令原文上下文（Call Flow、硬约束摘要、Opening Line 说明），对每个维度给出 1–5 分并提供思维链推理。各维度的 1–5 分通过"得分 ÷ 5 × 100"线性映射至 0–100 分，再按六维权重加权求和，得到最终 LLM 分（0–100 分）。系统为 LLM 评委设置了若干强制规则：每个维度的推理必须引用至少一个 [Tn] 轮次标记，无法引用具体轮次的维度得分上限为 2 分，确保评分有对话文本依据而非凭感觉给出。LLM 调用失败（超时或格式解析错误）时，系统自动降级为全维度 3 分的兜底分数，并标记 `degraded=True`；系统在降级前最多重试 2 次。

**加权合并（Aggregator）。** 两轨分数按"总分 = 规则分 × 0.40 + LLM 分 × 0.60"合并，结果截断至 [0, 100] 区间。若 `hard_fail` 为真，合并直接返回 0 分、等级 F，不执行加权计算。最终总分换算为五级等级：90 分及以上为 A，80 至 89 分为 B，70 至 79 分为 C，60 至 69 分为 D，60 分以下为 F。

---

### 4.3 总体评分维度与能力画像

LLM 评委分的六个维度共同构成被测 Bot 的能力画像，覆盖从流程执行到表达质量的完整侧面。

**流程遵循**（权重 25%）评价 Bot 是否按照指令规定的 Call Flow 顺序推进对话，能否在用户插话、追问后自然回归流程节点。1 分代表完全无序或大量跳步，3 分代表基本按流程但有 1–2 处小偏差，5 分代表完美遵循且对流程内的用户插话应对自然。该维度权重最高，因为流程遵循是通话质量的基础保障。

**话术合规**（权重 20%）评价 Bot 的话术是否符合指令规定的约束，包括字数限制（每轮不超过 30 字，Opening Line 豁免）、禁用词、表达风格等。系统特别设置了话术张力规则：偶发的 31–35 字且语义完整的回复，在此维度最高可给 4 分，自然度维度不重复扣分，避免因轻微超字而双重惩罚；只有明显超长或频繁违规才在此维度扣分。

**知识准确**（权重 20%）评价 Bot 在回答用户追问时引用的业务知识是否准确，是否存在捏造信息或引用错误规则的情况。1 分代表严重错误或捏造，3 分代表基本准确但有小瑕疵，5 分代表准确且引用恰当。

**挽留效果**（权重 15%）仅在对话进入异议或挽留节点（OBJECTION、F3\_RETAIN、OBJ\_FINAL）时适用，评价 Bot 的挽留策略是否得当。若对话中用户全程配合、从未拒绝，该维度固定给 4 分，不因未主动挽留而扣分，因为挽留场景本身未发生。

**边界处理**（权重 10%）评价 Bot 面对越界问题（用户提出超出任务范围的请求）时是否能守住授权边界，避免越权承诺，同时礼貌收束。

**自然度**（权重 10%）评价 Bot 的表达是否口语化、像真人，而非机械、生硬。该维度与话术合规维度相互独立，不因轻微字数超出而同步压低。

六维得分可在报告的 `dimension_scores` 字段中完整查看，`dimension_evidence` 则记录了每个维度对应的思维链推理、证据轮次列表和主要问题描述，形成对 Bot 能力的结构化画像，支持按维度做横向对比和弱项定位。

---

### 4.4 系统综合评分

单条执行计划的 `EvalReport` 只反映特定路径、特定 Persona 组合下的表现。`EvalRunner` 在运行完所有执行计划后，对当前数据集的全部报告做聚合统计，输出系统级综合评分。聚合结果包括：所有用例的平均分（`average_score`）、按路径 ID 分组的分项均分（反映各类对话路径下的表现差异）、按 Persona 类型分组的分项均分（反映面对不同用户行为时的能力差异）、六维 LLM 评分的跨用例均值（形成维度雷达图数据），以及等级分布统计（各等级的用例数量与比例）。

系统还维护运行元数据，包括被测 Bot 使用的模型名称（`bot_model`）、评委模型（`judge_temperature`）、并发数（`max_concurrent_dialogues`）、超时设置（`plan_timeout_sec`）和实际运行耗时，支持不同模型之间的横向对比分析。全部结果写入 `eval1/outputs/eval1_reports_{dataset_id}.json`，使用不同 Bot 提供方时会附加后缀区分（如 `_deepseek`）。

---

### 4.5 报告生成与解释

每条 `EvalReport` 包含以下主要字段，构成对单次仿真的完整解释。

基本标识与得分方面，`report_id` 唯一标识报告，`plan_id` 关联对应的执行计划，`path_id` 和 `persona_type` 说明本次仿真采用的路径和 Persona，`total_score` 为最终总分，`grade` 为对应等级，`rule_score` 和 `llm_score` 分别展示两轨原始分，`score_breakdown` 输出形如"规则分=85.0×0.40+LLM分=72.0×0.60 => 77.2 (C)"的可读得分构成字符串。

过程指标方面，`flow_adherence_rate` 记录路径节点覆盖率（实际覆盖节点数除以总节点数），`total_turns` 记录实际对话总轮数，`termination_reason` 记录对话终止原因（`hard_violation`、`hangup`、`goal_achieved`、`user_refused`、`max_turns` 之一）。

违规证据方面，`violations` 列表中每条记录携带触发违规的轮次编号（`turn_index`）、约束标识（`constraint_id`）、Bot 原始话语（`bot_utterance`）、违规说明（`explanation`）和扣分值（`deduction`），支持从最终总分逐步追溯到具体轮次的具体问题。

LLM 评分侧，`dimension_scores` 存储六维得分（0–100 分），`dimension_evidence` 为每个维度提供思维链推理文本、引用的 [Tn] 轮次列表和主要问题关键词，`judge_comment` 是 LLM 评委的总体评语，`top_improvement` 是最重要的改进建议。`improvement_suggestions` 列表汇总了规则层和 LLM 层生成的所有改进方向，以及动作检测或评分降级时的人工复核提醒（`needs_human_review`）。

## 五、三层联动关系

三个层次之间通过明确的数据结构接口串联，形成一条完整的评测流水线。Layer1 将自然语言指令解析为规则知识图谱，枚举出覆盖各类场景的路径集合，并与全部六种 Persona 做笛卡尔积，生成执行计划列表（`ExecutionPlan`）。Layer2 以 LangGraph StateGraph 为引擎，按执行计划逐条驱动多轮对话仿真，输出包含完整对话历史、实时违规记录、路径覆盖率和终止原因的仿真结果（`DialogueTrace`）。Layer3 接收仿真结果，通过规则扣分和 LLM 多维评估两条轨道计算分数，汇总为带等级和可解释证据的评估报告（`EvalReport`），最终写入 `eval1/outputs/` 目录下的 JSON 文件。

三层之间的接口类型（`ParsedInstruction`、`EnumeratedPath`、`ExecutionPlan`、`DialogueTrace`、`EvalReport`）均有明确的 Pydantic 数据模型定义，每层可独立运行和调试，前端工作台为每层提供专属的可视化界面，支持人工逐层审查和干预。
